## SQLAlchemy 继承与会话机制简述（简版）

- 运行原理（简要）：
  - 联表继承（joined-table inheritance）：一个对象映射到多张表（父/子表）。对“实例删除”时，ORM 会按继承映射自动生成多表 `DELETE`，先删子表再删父表。
  - 关系级联（ORM 层）：在父侧关系上配置 `cascade='all, delete-orphan'`，删除父对象时，已加载的子集合（如 `IndexAlias`）会一并删除，无需数据库 `ondelete=CASCADE`。
  - 会话身份映射（Identity Map）：同一 Session 内，同一主键只对应一个 Python 对象实例且类型不变。多态转换后如果立即查询，仍可能返回旧类型；需驱逐旧对象或换新会话。

- 推荐用法：
  - 单条删除（继承+别名级联）：
    ```python
    from sqlalchemy.orm import selectinload

    base = db.session.query(IndexBase)\
        .options(selectinload(IndexBase.aliases))\
        .filter(IndexBase.id == index_id).first()
    if not base: raise NotFound
    db.session.delete(base)  # ORM 自动处理子表与已加载的别名集合
    db.session.commit()
    ```
  - 多态转换后正确读取：
    ```python
    service.update_with_polymorphic_conversion(id, data)
    db.session.commit()
    db.session.expunge(old_instance)  # 驱逐旧对象，避免返回缓存的旧类型
    obj = db.session.query(StockIndex).filter_by(id=id).first()
    ```

- 何时需要“手动分层删除”或数据库级联：
  - 批量条件删除（`Query.delete()`/原生 SQL）不会触发 ORM 级联与继承多表删除，需按“先子后父”手动处理，或在数据库外键上启用 `ondelete=CASCADE`（这会涉及迁移脚本与引擎支持）。

以上做法不改变数据库结构，不会生成迁移脚本；仅影响 ORM 行为，使单条删除与转换后的读取更符合预期、实现更简洁。

## SQLAlchemy 多态转换（Polymorphic Conversion）中的陷阱与解决方案

### 问题描述

在使用 SQLAlchemy 进行继承模型的**多态类型转换**时（例如，将一个 `Asset` 类型的实例转换为 `AssetFund` 类型），即使后台数据库中的多态鉴别器字段（如 `asset_subtype`）已经被正确更新，在同一个 Session 中立即执行对新类型的查询，返回的仍然是**旧类型**的 Python 对象实例。

**场景复现:**

1. 在一个 Session 中加载一个基类 `Asset` 的实例 `original_asset`。
2. 调用服务执行多态转换逻辑，该逻辑成功 `UPDATE` 了数据库中 `asset` 表的 `asset_subtype` 字段为 `'fund'`，并在 `asset_fund` 子表中插入了数据。
3. 在同一个 Session 中执行查询 `session.query(AssetFund).filter(AssetFund.id == original_asset.id).first()`。
4. **预期结果**: 得到一个 `AssetFund` 类型的实例。
5. **实际结果**: 仍然得到最初加载的那个 `Asset` 类型的实例 `original_asset`。

### 根本原因：Session 的身份映射（Identity Map）机制

这个问题的核心是 SQLAlchemy 为了保证数据一致性和性能而设计的 **Identity Map** 机制。

1. **对象身份唯一性**: 在一个 Session 的生命周期内，对于数据库中的同一行数据（由主键标识），只允许存在唯一一个与之对应的 Python 对象实例。
2. **类型不可变性**: 当一个对象被加载到 Session 中时，它的 Python 类型（`__class__`）就被确定下来了。Session 不会，也不能在后续的操作中改变一个已存在于内存中的对象的类型。
3. **缓存优先**: 当执行查询时，SQLAlchemy 会首先检查 Identity Map。如果发现目标主键的对象已经存在于 Session 缓存中，它会**立即返回该缓存对象**，并**跳过**对数据库的 `SELECT` 查询，以遵循“身份唯一性”原则。

因此，在上述场景中，即使数据库的数据已经更新为 `AssetFund`，但由于最初的 `Asset` 对象仍在 Session 的 Identity Map 中，SQLAlchemy 会直接返回这个旧对象，导致查询结果不符合预期。

### 无效的解决方案

* `session.expire(instance)`: 这个方法只会将实例的属性标记为“已过期”，使得下次访问属性时会重新从数据库加载。但是，它**不会改变对象本身的 Python 类型**，也**不会将其从 Identity Map 中移除**。因此，即使数据被重新加载，也只是填充到那个旧的、类型错误的对象里。

### 正确的解决方案

要解决这个问题，必须在查询新类型之前，将旧的、类型已过时的对象从 Session 的 Identity Map 中移除。

#### 方案一：`session.expunge(instance)` (推荐)

这是最直接和常用的方法。`expunge` 会将指定的对象实例从 Session 中彻底驱逐出去，解除 Session 对它的所有跟踪。

```python
# 1. 执行多态转换并提交
service.update_with_polymorphic_conversion(...)
session.commit()

# 2. 从 Session 中驱逐旧的实例
# original_asset 是转换前加载到内存中的那个对象
session.expunge(original_asset)

# 3. 重新查询
# 此时 Identity Map 中已没有该对象，查询将访问数据库
# 并根据新的 asset_subtype='fund' 正确构建一个 AssetFund 实例
new_fund = session.query(AssetFund).filter(AssetFund.id == original_asset.id).first()

# new_fund 现在是正确的 AssetFund 类型
assert isinstance(new_fund, AssetFund)
```

#### 方案二：使用全新的 Session (测试中的最佳实践)

在真实的 Web 应用中，更新和后续的读取操作通常发生在不同的请求中，也就是不同的 Session 里。在测试中模拟这种行为是更可靠的方法。

```python
# --- 事务 A ---
session_a = Session()
# 加载对象并执行转换
# ...
session_a.commit()
session_a.close()

# --- 事务 B (模拟下一次请求) ---
session_b = Session()
# 在一个全新的、干净的 Session 中查询
new_fund = session_b.query(AssetFund).filter(AssetFund.id == an_id).first()
session_b.close()

# new_fund 将会是正确的类型
```

这个方法最能反映真实的应用场景，能避免很多与 Session 状态相关的测试问题。
