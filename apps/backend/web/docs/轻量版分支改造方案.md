# Snowball 轻量版分支改造方案

> 归档说明
>
> 当前长期版本见 [docs/architecture/lite-boundary.md](../../../../docs/architecture/lite-boundary.md)。
> 这份文档保留阶段过程和历史背景，后续正式维护以根目录 `docs/` 下的长期文档为准。

## 当前定位

这份文档现在不再作为“待执行的大规划”，而是作为轻量版改造的总览文档。

重点只保留三类信息：

- 轻量化为什么要做
- 各阶段到底做了什么
- 当前已经走到哪一步、还没走到哪一步

当前项目的轻量化目标，始终不是“只把 MySQL 换成 SQLite”，而是整体收缩系统边界，让项目更适合：

- 单人使用
- 本地运行
- 单机、低并发
- 更少的常驻依赖

## 轻量版目标

轻量版希望保留的核心能力：

- 资产基础管理
- 交易记录管理
- 基础分析能力
- `xalpha` / DataBox 拉数
- 必要的查询类 API

轻量版希望优先降级或移除的部分：

- Redis 作为必选依赖
- Dramatiq 常驻 worker
- APScheduler 的数据库持久化能力
- Flask-Profiler
- 多环境部署下那套较重的运行前提

## 阶段 0：建立轻量版分支

这一阶段的核心工作，是把轻量化尝试和原主线隔开，先允许试错。

本阶段完成的事情：

- 明确轻量版改造不直接在原主线硬改
- 先用独立分支和独立仓库基线承接 spike 结果
- 后续把阶段性结果沉淀为 `snowball-lite` 仓库

这一阶段的价值，不在于功能，而在于给后续轻量化改造留出独立空间。

- lite 模式下可以不强依赖 Redis、Dramatiq、持久化 scheduler、profiler
- 建立 SQLite 最小可用链路
- 验证 `xalpha` / DataBox 的最小兼容路径
- 补了最小接口 smoke 验证

## 当前结论

到现在为止，轻量化已经完成了很重要的一段，但还没有“全部完成”。

已经完成的部分：

- lite 启动骨架已经稳定
- 核心可选基础设施已经被拆成可关闭
- SQLite 最小可用链路已经稳定
- 真实 DataBox 验证已经补过
- 一批高风险 SQLite 模型已经验证过
- 一批更业务化的 SQLite 集成链路已经验证过
- lite 已经有自己的 migration baseline

还没有完成的部分：

- 整个 `web/webtest` 还没有都脱离 MySQL
- `migrations_snowball_lite` 目前还是受控范围内的 baseline，不是完整 lite schema
- 仓库里剩余 MySQL 专有路径还没有全部系统清理
- 还不能把当前状态认定为“全仓库 SQLite 迁移完成”

## 阶段 5：收口 lite 的 SQLite 初始化和运行时

这一阶段解决的是“lite 虽然越来越能跑，但还没有形成正式主线口径”。

本阶段完成的事情：

- 已把 lite fixture 和最小链路验证收口到 `bootstrap_lite_database(...)`，lite 主线不再靠 `db.create_all()` 兜底

- 已把 `/system/token` 和 `/api/enums/versions` 在 lite 下收口成显式不支持，而不是隐式 500

## 阶段 6：零 MySQL 环境收口验收

这一阶段解决的是“前面都做完了以后，怎么证明 lite 主线真的已经脱离 MySQL”。

阶段 6 的主目标是：

- 在没有 MySQL server 的前提下，完成 lite 的启动、初始化、业务验证和最终收口
- 给“lite 主线已经可以纯 SQLite 运行”这个结论补上正式证据

本阶段建议完成的事情：

- 在零 MySQL 环境下启动 lite
- 在零 MySQL 环境下执行 lite 核心回归和业务验收
- 对这一阶段的实现结果做正式代码审查
- 对审查问题做修复、复验和再审查
- 输出最终阶段结论和剩余边界说明

阶段六归档文档：

- `apps/backend/web/docs/desc/lite_stage6/00_stage6_overview.md`
- `apps/backend/web/docs/desc/lite_stage6/01_zero_mysql_startup_and_bootstrap_validation.md`
- `apps/backend/web/docs/desc/lite_stage6/02_zero_mysql_core_business_validation.md`
- `apps/backend/web/docs/desc/lite_stage6/03_zero_mysql_blocker_cleanup.md`
- `apps/backend/web/docs/desc/lite_stage6/04_stage6_final_review_and_rereview.md`
- `apps/backend/web/docs/desc/lite_stage6/05_stage6_acceptance_and_decision.md`

这一阶段要回答的关键问题是：

- lite 是否已经真正不依赖 MySQL
- lite 的核心业务操作是否已经都能在纯 SQLite 环境下完成
- 现在是否可以把 lite 主线认定为“完成脱离 MySQL”

阶段 6 的验收标准：

- 没有 MySQL server 也能完成 lite 启动和初始化
- lite 核心业务操作能在纯 SQLite 环境下完成
- 最终代码审查后没有中等或以上问题残留
- 可以明确输出“lite 主线已脱离 MySQL”或“仍有 blocker”的结论

阶段 6 的非目标：

- 不代替全仓库最终 gate
- 不等于全仓库 SQLite 迁移完成
- 不要求非 lite 配置放弃 MySQL 兼容

## 后续方向

如果后面继续推进，建议按阶段 4、阶段 5、阶段 6 这条线继续往前做。

当前最优先的仍然是：

1. 先做阶段 6，在零 MySQL 环境下完成集中验收
2. 不要回头把阶段 4、阶段 5 已经收口的任务再当成待开始事项重复推进
3. 不要把阶段 6 的目标扩成“全仓库 SQLite 化”

不建议现在做的事：

- 不建议立刻把全部旧测试统一切到 SQLite
- 不建议现在重写 dev/stg/test/prod 的完整历史迁移
- 不建议把当前结果误读成“轻量化已经全部完成”
- 不建议把“lite 主线脱离 MySQL”和“全仓库 SQLite 化”混成一件事
