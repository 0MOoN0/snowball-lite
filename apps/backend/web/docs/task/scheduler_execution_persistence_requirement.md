## 0. 先说当前结论
当前真正要解决的问题不是“outbox 太频繁”，而是共享 listener 把不同性质的任务都当成同一种运行事件处理。

现在 `tb_apscheduler_log` 一张 append-only 表同时承担三类需求：
1. 任务最新状态展示
2. 手动触发后的短时防重
3. 错误排障和异常日报

完整需求的完成口径是：
- 系统具备任务级执行持久化策略
- lite 主线已注册任务都完成策略归类
- 高频轮询任务不再因为空轮询持续制造无效成功日志
- 现有依赖方仍保留最小可用能力

## 1. 问题定义
当前 listener 对 `SUBMITTED`、`EXECUTED`、`ERROR`、`MISSED` 使用统一落库规则，这会带来三件事：
1. 高频轮询任务的空执行被当成完整运行历史写入
2. “最近一次有意义执行”和“最近一次被调度”在库里没有区分
3. 依赖方继续拿一张历史表兼做状态表，放大写入量和语义混乱

根因不是没有清理任务，而是系统没有表达“某个任务的某次运行是否值得持久化”的能力。

## 2. 完成定义
这份需求完成后，必须同时满足下面几条：
- scheduler listener 在写 `tb_apscheduler_log` 前，会按解析后的 `job_id` 命中执行持久化策略
- lite 主线常驻任务都完成一次策略归类，结果要么保留 `full`，要么切到更收敛的策略
- 高频轮询任务的空轮询不会再持续写入成功记录
- `ERROR` / `MISSED` 仍然保留稳定落库能力
- 手动触发的一次性 job 仍保留当前防重能力
- 任务列表、任务日志详情、异常日报继续可用，但不再假设“每次调度都一定有一条新成功记录”

## 3. 范围
本需求覆盖下面这些内容：
- `apps/backend/web/scheduler/__init__.py` 的共享 listener
- 代码侧的任务执行持久化策略模型和注册方式
- lite 主线已注册 scheduler 任务的策略归类
- 首批高频轮询任务的接入和验证
- 对现有查询接口与日报逻辑的兼容说明

## 4. 非目标
本需求不把下面这些事情算进完成条件：
- 不把 `tb_apscheduler_log` 直接重构成“状态表 + 事件表”
- 不做运行时动态配置化策略后台
- 不清理历史 `tb_apscheduler_log` 记录
- 不把 `tb_notification_outbox` 的保留期清理并入本需求
- 不承诺所有历史 MySQL 环境都引入额外差异化策略

## 5. 方案
采用“任务级执行持久化策略”方案。

策略模型至少支持三类：
1. `full`：保持当前行为，`SUBMITTED`、`EXECUTED`、`ERROR`、`MISSED` 按现有规则持久化。
2. `signal_only`：`ERROR`、`MISSED` 仍持久化，`EXECUTED` 只有在本次运行带来业务信号时才持久化，适用于高频轮询任务。
3. `error_only`：只保留错误和错过执行，成功运行不写库，适用于只关心异常的维护型任务。

策略归类规则：
- 默认策略仍是 `full`
- 只有显式评估过“成功执行是否有业务信号”的任务，才能切到 `signal_only`
- 如果任务没有明确的成功信号判定方法，就不能直接切 `signal_only`
- 手动触发防重依赖的 `SUBMITTED` 记录不能因为策略收口被误伤

“业务信号”不是“函数正常返回”，而是“这次运行确实处理了业务数据或产生了可追踪结果”。
- `signal_only` 任务必须提供可测试的信号判定方法
- 没有信号判定就继续保留 `full`
- `ERROR` 和 `MISSED` 不参与信号判断，默认都保留

## 6. 数据与存储影响
- `tb_apscheduler_log` 继续保留为运行观测表
- 本需求优先改变写入规则，不要求改表结构
- `tb_notification_outbox` 继续是业务 outbox，不承担 scheduler 运行观测职责

数据归属边界：
- 业务事件数据归 `tb_notification_outbox` 等业务表
- scheduler 运行观测归 `tb_apscheduler_log`

## 7. API 与交互影响
- `/scheduler/jobs` 仍可展示任务和最近一次已持久化记录
- `/scheduler/job_log/<job_id>` 继续返回最近一条已持久化记录
- 手动触发接口继续依赖最近 `SUBMITTED` 记录做短时防重
- 异常日报继续只关注 `ERROR` / `MISSED`

这里要明确：
- “最近一次持久化记录”不再等于“最近一次调度时间”
- 对 `signal_only` 或 `error_only` 任务，空轮询成功不会刷新最近记录

## 8. 生命周期与旧数据处理
本需求只约束新写入规则，不处理历史回填。
1. 历史 `tb_apscheduler_log` 保持原状
2. 新策略只影响需求落地后的新增运行记录
3. 如果后续要做保留期清理、归档或物理删除，另开任务

## 9. 分阶段计划
- 阶段 1：交付策略基座，接入 `AsyncTaskScheduler.consume_notification_outbox`，验证 `signal_only` 能拦住空轮询成功记录
- 阶段 2：盘点 lite 主线其它常驻任务，为适合的任务补 `signal_only` 或 `error_only`，不适合的继续走 `full`
- 阶段 3：补文档、查询口径说明和必要测试，并评估是否单独立项做日志保留期清理

## 10. 验收标准
- listener 能按 `job_id` 命中策略并稳定工作
- lite 主线常驻任务都有策略归类结果
- `signal_only` 任务空轮询不再新增成功记录
- `ERROR` / `MISSED` 的写库能力不回归
- 手动触发防重逻辑不回归
- 现有任务列表和异常日报在新口径下仍可用

## 11. 风险点
- 如果只做首批任务试点，不继续完成任务归类，需求只能算阶段完成，不能算总体完成
- 如果把“正常返回”错当成“有业务信号”，高频任务的写入量不会真正降下来
- 如果依赖方仍按“最近一次调度必有日志”理解新策略，会误读旧时间戳

## 12. 后续边界
下面这些问题需要明确保留为后续事项，不能混进本需求验收：
1. `tb_apscheduler_log` 是否拆成状态表和事件表
2. 历史日志保留期和物理清理
3. `tb_notification_outbox` 的成功/失败记录保留期
