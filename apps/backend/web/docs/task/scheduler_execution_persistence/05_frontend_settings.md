# scheduler 前端单任务策略设置任务
- [ ] 在 scheduler 页面展示任务当前生效策略
- [ ] 提供单任务策略设置入口和表单交互
- [ ] 只允许选择后端返回的 `supportedPolicies`
- [ ] 保存后刷新当前任务行数据
- [ ] 明确“默认值 / 覆盖值 / 生效值”的展示口径

## 任务状态
- 类型：子任务文档
- 父任务：`/Users/leon/projects/snowball-lite/apps/backend/web/docs/task/scheduler_execution_persistence/requirement.md`
- 依赖：`/Users/leon/projects/snowball-lite/apps/backend/web/docs/task/scheduler_execution_persistence/04_persistence_api.md`
- 状态：待开始
- 目标：把任务策略覆盖开放到前端，但不把 scheduler 页面变成“编辑 cron / kwargs”的全功能任务编辑器

## 1. 问题定义
现有 scheduler 页面只有查看、运行、暂停、恢复和日志查看。

虽然页面上有“编辑任务”按钮，但当前弹窗没有真实保存能力，也没有任务策略字段。若直接把“编辑任务”继续做大，会把“策略设置”和“任务定义编辑”混成一件事。

## 2. 完成定义
本任务完成后，必须同时满足下面几条：
- 用户能在前端看到每个任务的默认策略、当前生效策略和来源
- 用户能对支持切换的任务单独设置策略
- 表单只能选后端返回的 `supportedPolicies`
- 保存成功后，当前任务行会刷新到最新值
- 不支持切换的任务会明确展示为只读

## 3. 范围
本任务只覆盖下面这些内容：
- `apps/frontend/src/views/Snow/Setting/Scheduler/`
- 对应 scheduler API 封装和类型定义
- 任务列表页上的策略展示和单任务设置对话框

## 4. 非目标
本任务不做下面这些事情：
- 不提供 cron、trigger、args、kwargs 的通用编辑能力
- 不做批量策略编辑页面
- 不直接编辑原始 `system_settings` JSON
- 不改后端默认策略归类逻辑

## 5. 选定方案
采用“任务列表页展示 + 单任务设置弹窗”的方案。

UI 规则：
1. 在任务列表增加策略展示列
2. 增加“设置策略”入口，和现有“查看日志 / 运行 / 暂停 / 恢复”分开
3. 弹窗只编辑策略，不编辑 cron 和 kwargs
4. 页面要同时展示：
   - 默认策略
   - 当前生效策略
   - 来源：`default` 或 `override`

现有“编辑任务”按钮如果继续保留，必须改名或改行为，避免误导用户认为可以改调度定义。

## 6. 数据与交互影响
- 前端不直接拼装 `system_settings` JSON
- 前端通过 scheduler 专属接口查询和更新策略
- 页面本地状态需要兼容“只读任务”和“可配置任务”

## 7. 生命周期与旧数据处理
前端只处理当前可见任务：
1. 页面加载时读取后端返回的生效策略
2. 用户保存后，以后端返回值为准刷新当前行
3. 如果任务不再存在，前端不负责清理旧覆盖值

## 8. 验收标准
- 任务列表能展示策略字段
- 用户能为支持切换的任务单独设置策略
- 不支持切换的任务不会出现可提交的非法选项
- 保存后当前行能看到更新后的 `effectivePolicy` 和 `policySource`
- 页面不再把“编辑任务”误导成“编辑调度定义”

## 9. 验证要求
- 前端 API 类型和请求测试
- 页面交互测试或至少关键逻辑断言
- 非法选项保护和保存后刷新验证

## 10. 风险点
- 如果继续复用当前“编辑任务”弹窗，会让用户以为能改触发器和参数
- 如果不展示 `policySource`，用户会分不清自己看到的是默认值还是覆盖值
- 如果前端缓存不刷新，保存后页面会出现旧值

## 11. 后续动作
本任务完成后，如果还要继续做前端增强，另开任务处理：
1. 批量策略编辑
2. 策略变更历史
3. 历史覆盖值清理提示
