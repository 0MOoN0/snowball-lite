## 1. 原始目标

这一步只做运行对齐：

- 前端能请求当前后端
- 核心业务页面能读到真实或测试数据
- 把前后端的已知差异明确记录下来

## 2. 初始已知情况

`snow_view` 里的业务模块和后端总体是对得上的：

- 资产
- 记录
- 指数
- 网格分析
- 通知
- 系统设置

但仍然存在一些不能直接假设“搬进来就全通”的点：

- 前端默认代理和环境变量仍偏向旧 dev 环境
- lite 下不保证 scheduler 主链路
- lite 下 `/system/token` 不属于主线能力
- 前端里仍保留部分旧接口风格

## 3. 原始范围

### 3.1 要处理的内容

- 前端 Vite 代理配置
- API base path 配置
- `.env.*` 工作区化
- 前后端本地端口对齐
- lite / dev 两种口径的可运行说明

### 3.2 最低联调范围

建议优先验这几组：

- 资产列表 / 详情
- 记录列表 / 详情
- 指数列表 / 详情
- grid / grid-type 分析结果
- 通知列表 / 详情
- 系统设置列表

## 4. 非目标

- 这一步不要求前端所有页面全量通过
- 这一步不强行补完后端所有旧接口
- 这一步不处理后端物理迁移

## 5. 关键改造点

### 5.1 先跑主链路，不追求一次全通

先把核心读路径跑通，再决定哪些页面需要补接口、哪些页面在 lite 下先降级。

### 5.2 lite 与 dev 要分开说明

这一步必须明确两件事：

- 哪些页面在 lite 下就是可跑的
- 哪些页面依赖旧 dev / scheduler / token 路径

不要把这两套口径混成“前端都支持”。

## 6. 原始验收标准

- `apps/frontend` 可以在主仓里本地启动
- 前端能稳定请求当前后端
- 至少一组核心页面完成联调
- 已知不兼容点有文档记录
- lite / dev 运行说明能让别人照着起

## 7. 初始风险点

- 如果把“接口适配”和“前端重构”混在一起，联调周期会失控
- 如果不区分 lite 和 dev，后续验收时会反复误判
