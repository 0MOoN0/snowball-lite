# 任务 4：最小接口冒烟与结论输出

## 目标

在轻量模式下完成最小接口级验证，并输出是否继续推进轻量版的结论。

本任务是本轮 spike 的收口任务，用于判断：

- 轻量模式是否已经具备继续推进价值
- 后续更适合留在主仓库，还是转向独立项目

## 任务范围

- 选 2～3 条核心接口做冒烟验证
- 调整最小测试或手工验证脚本
- 记录 blocker、成本和下一步建议
- 将结果回写到轻量版文档

## 关键文件

- `web/webtest/conftest.py`
- 核心 router / service 文件
- `web/docs/轻量版分支改造方案.md`
- `web/docs/desc/lite_migration/stage1/*.md`

## 推荐验证接口

建议优先选这 3 类：

- 资产查询类接口
- 记录读取或写入类接口
- 依赖 DataBox 的一个最小读取接口

## 验证方式

可接受两种方式：

- 最小 pytest / smoke test
- 手动脚本 + 日志验证

本轮优先级是“快速获得可信结论”，不是“测试体系重建”。

## 执行步骤

1. 选定 2～3 条关键接口
2. 为轻量模式准备最小 fixture 或手工验证数据
3. 验证应用启动、数据写入、接口返回
4. 记录通过项和 blocker
5. 回写文档并给出结论

## 验收标准

- 至少 2 条关键接口可在轻量模式下返回
- 至少 1 条接口能覆盖数据库读写
- 至少 1 条接口能覆盖 DataBox 或缓存链路
- 形成明确结论：继续 / 暂停 / 拆分项目

## 非目标

- 不要求全量接口可用
- 不要求完整自动化回归通过
- 不要求清理所有历史测试债务

## 结论模板

文档收尾时建议固定输出：

- `结论`
- `已验证通过`
- `主要 blocker`
- `下一阶段工作量`
- `建议下一步`

## 风险提示

- 如果接口选择过多，本任务会变成“伪全量验证”
- 如果没有 blocker 记录，后续决策容易失真

## 任务完成产物

- 一份最小 smoke 验证结果
- 一份 blocker 清单
- 一份下一阶段建议

## 本轮结果

### 结论

- 继续。

### 已验证通过

- `GET /api/asset/list/` 可以从 SQLite 返回种子资产数据。
- `POST /api/record` + `GET /api/record/<id>` 可以完成最小数据库读写闭环。
- `GET /token_test/result` 已覆盖到 `TokenTest -> DataBox.get_rt -> XaDataAdapter.get_rt` 这条最小运行路径，只是把更底层的 `xalpha.get_rt` 打桩了。

### 主要 blocker

- 当前 smoke 仍不等于真实 token、真实网络、真实第三方数据源稳定可用。
- 接口级 smoke 还没有把“真实缓存写入再读取”放进同一条 HTTP 验证链路里。
- 这轮验证的是轻量模式最小闭环，不代表所有历史模型和迁移路径都已经兼容 SQLite。

### 建议下一步

- 继续留在当前仓库推进，不建议现在拆独立项目。
- 下一阶段优先补更真实的 DataBox / 第三方数据验证。
- 再扩 1 到 2 条高频业务接口，确认 lite 不是只停留在最小 happy path。
