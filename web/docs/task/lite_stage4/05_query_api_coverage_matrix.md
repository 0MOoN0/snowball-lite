# 任务 5：必要查询类 API 覆盖清单

## 当前状态

- 状态：待开始
- 任务定位：收口第四阶段的对外查询面，并输出 lite 能力覆盖清单

## 目标

把前面几个业务任务里的查询能力收成一套可追踪的对外接口覆盖面，而不是只留下零散测试。

这一任务要回答的问题是：

- lite 对外暴露的关键查询类 API，到底已经覆盖了哪些
- 还有哪些能力只是局部 smoke、手工验收，或者还没纳入

## 为什么要单独做这一项

任务 1 到任务 4 更偏“按业务域补闭环”。

但阶段 4 还有一个明确要求：

- 要形成一份“已覆盖 / 未覆盖”的能力清单

这件事如果不单独做，最后大概率会漏掉，或者只能靠口头总结。

## 建议范围

- 资产查询类接口
- 交易记录查询类接口
- 分析结果查询类接口
- `token_test` 这类系统查询入口
- 对应测试文件、测试类型和运行命令的映射整理

## 建议输出内容

- 一份 lite 查询面能力清单
- 每条能力对应的：
  - 入口接口或服务
  - 自动化测试位置
  - 是否需要手工验收
  - 当前覆盖状态
- 一条固定的阶段 4 查询面回归命令

## 建议落点

- 建议新增测试聚合文件：`tests/test_lite_stage4_query_api_matrix.py`
- 建议新增阶段总结文档草稿：`web/docs/desc/lite_stage4/` 下的覆盖清单文档

## 建议实施方式

1. 先收集任务 1 到任务 4 实际覆盖到的查询面
2. 再补最明显缺口
3. 最后整理成一份能直接看懂的覆盖矩阵

## 验收标准

- 有一份清楚的“已覆盖 / 未覆盖”能力清单
- 关键查询类 API 都能映射到具体测试或手工验收入口
- 有一条固定命令能回归阶段 4 选中的查询面验证

## 非目标

- 不代替阶段 4 最终验收报告
- 不要求把所有非 lite 范围接口都整理进去
- 不要求现在处理阶段 5 和阶段 6 的内容
