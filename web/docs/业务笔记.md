# 金融数据模型设计笔记

本文档旨在记录项目核心数据模型的设计思路、具体实现与未来扩展方向。

---

## 一、核心设计原则

### 1.1 组合优于继承：`Index` 与 `Asset` 的关系

在系统设计初期，我们面临一个关键问题：**指数（Index）是否应该继承资产（Asset）？**

我们的答案是：**不应该**。这个决策基于“组合优于继承”的核心原则：

1.  **继承代表“Is-A”关系**：资产（Asset）是一个可以被拥有、交易、并计算价值和数量的实体。如果 `Index` 继承 `Asset`，意味着“指数是一种资产”，这在业务逻辑上是不成立的。
2.  **指数是“基准”而非“资产”**：指数（如沪深300）是一个衡量市场表现的度量工具，不能直接交易。强制继承会破坏设计的清晰性。
3.  **结论：采用“组合”**：`Index` 应被设计成一个独立的类。而 `IndexFund`（指数基金）作为 `Asset` 的子类，在其内部**包含**一个 `Index` 对象作为属性，清晰地反映了“is-a”和“has-a”的关系。

### 1.2 资产继承策略：基金模型的层次化设计

与指数不同，基金、股票等金融工具天然就是“资产”的一种。因此，采用**继承**来构建资产模型体系是合理且高效的。

我们推荐使用一个两层继承结构来设计基金模型：

1.  **`Asset -> Fund` (通用基金层)**：首先，创建一个通用的 `Fund` 模型，它继承自 `Asset`。此模型包含所有基金类型（无论股票、债券、货币）都通用的属性，如基金经理、成立日期、管理公司等。
2.  **`Fund -> ETF / MutualFund` (具体基金层)**：然后，创建更具体的基金模型，如 `ETF`、`MutualFund`（共同基金）等，它们都继承自 `Fund` 模型。这些子类模型用于定义特定类型基金独有的属性，例如 `ETF` 的 `tracking_index`（跟踪指数）和 `expense_ratio`（管理费率）。

这种 `Asset -> Fund -> 具体基金` 的继承链，不仅结构清晰、扩展性强，而且与项目现有的多态设计（`polymorphic_on`）完全兼容。

## 二、核心数据模型定义

### 2.1 指数模型 (Index)

`Index` 模型被定义为一个独立的信息实体，其核心职责是提供关于市场基准的数据。

#### 类定义
```python
from datetime import date
from typing import Optional

class Index:
    """
    表示一个金融指数，作为市场表现的基准。
    它不是一个可交易的资产，而是一个数据和信息的集合。
    """
    def __init__(self,
                 symbol: str,
                 name: str,
                 market: str,
                 publisher: Optional[str] = None,
                 base_date: Optional[date] = None,
                 base_value: Optional[float] = None,
                 description: Optional[str] = None):
        self.symbol = symbol
        self.name = name
        self.market = market
        self.publisher = publisher
        self.base_date = base_date
        self.base_value = base_value
        self.description = description

    def __repr__(self) -> str:
        return f"<Index symbol='{self.symbol}', name='{self.name}'>"
```

#### 字段分析

*   **核心字段**: `symbol` (主键), `name` (UI展示), `market` (分类)。
*   **扩展字段**: `publisher` (发布机构), `description` (背景知识)。
*   **高级字段**: `base_date` (基准日期), `base_value` (基准点数)。

### 2.2 通用行情模型 (KLine / OHLCV)

指数的点数变化与资产（如ETF、股票）的价格变化，在数据结构上高度同构，因此可以使用一个通用的 `KLine` 模型来表示其历史行情。

#### 类定义
```python
from dataclasses import dataclass
from datetime import date

@dataclass
class KLine:
    """
    通用的K线数据模型，用于表示任何金融工具的每日行情数据。
    """
    symbol: str       # 关联到 Index.symbol 或 Asset.code
    timestamp: date   # 数据对应的日期
    open: float       # 开盘价/点位
    high: float       # 最高价/点位
    low: float        # 最低价/点位
    close: float      # 收盘价/点位
    volume: int       # 成交量
    turnover: float   # 成交额
```
**注意**：虽然数据结构相同，但在业务逻辑中必须明确区分**指数的点数 (Points)** 和**资产的价格 (Currency)**。

### 2.3 资产与基金模型 (Asset & Fund Models)

资产模型是所有可交易金融工具的基石。我们采用分层继承策略，以 `Asset` 为顶层基类，向下逐层细化。

#### 2.3.1 顶层基类 (Asset)

`Asset` 模型定义了所有资产共有的基础属性，并通过 SQLAlchemy 的多态机制，根据 `asset_subtype` 字段动态实例化对应的子类。

```python
# 说明：这是一个简化的示意模型，实际字段可能更复杂
class Asset(Base):
    __tablename__ = 'assets'
    id = Column(Integer, primary_key=True)
    asset_code = Column(String(32), unique=True, nullable=False)
    asset_name = Column(String(64))
    asset_type = Column(String(32))  # e.g., 'fund', 'stock', 'bond'
    asset_subtype = Column(String(50)) # e.g., 'etf', 'mutual_fund'

    __mapper_args__ = {
        'polymorphic_identity': 'asset',
        'polymorphic_on': asset_subtype
    }
```

#### 2.3.2 通用基金模型 (Fund)

`Fund` 模型继承自 `Asset`，用于封装各类基金（如ETF、共同基金）的通用属性，如基金经理、管理公司等。

```python
# 说明：这是一个简化的示意模型
class Fund(Asset):
    __tablename__ = 'funds'
    id = Column(Integer, ForeignKey('assets.id'), primary_key=True)
    manager = Column(String(128))
    company = Column(String(128))
    inception_date = Column(Date)

    __mapper_args__ = {
        'polymorphic_identity': 'fund', # 实际应用中会根据 asset_type 进一步细分
    }
```

#### 2.3.3 具体基金模型 (ETF)

`ETF` 模型作为 `Fund` 的子类，增加了交易所交易基金特有的属性，如跟踪指数和管理费率。

```python
# 说明：这是一个简化的示意模型
class ETF(Fund):
    __tablename__ = 'etfs'
    id = Column(Integer, ForeignKey('funds.id'), primary_key=True)
    tracking_index_symbol = Column(String(32)) # 关联到 Index.symbol
    expense_ratio = Column(Float)

    __mapper_args__ = {
        'polymorphic_identity': 'etf',
    }
```

## 三、进阶设计与挑战

### 3.1 挑战：指数的多代码（别名）问题

**问题描述**：同一个指数在不同数据源下有多个代码（别名），如 `000300.SH`, `sh000300`, `CSI300`。

**最佳实践**：采用“规范代码 + 别名表”的数据库设计。

1.  **`Indexes` (指数主表)**: `symbol` (主键, 规范代码), `name`, `market` 等。
2.  **`IndexAliases` (指数别名表)**: `alias` (主键, 别名), `canonical_symbol` (外键, 指向 `Indexes.symbol`)。

**查询流程**：优先查询主表，失败则查询别名表，找到规范代码后再查询主表。

### 3.2 扩展性：应对不同类型的指数

**场景**：当需要处理不同类别指数的特有属性时（如债券指数的“平均久期”）。

**时机**：当代码中出现大量 `if/else` 判断指数类型时，就是引入派生设计的重构信号。

**示例**：
*   `BaseIndex` (通用字段)
    *   `StockIndex(BaseIndex)`: 增加 `constituent_count` 属性。
    *   `BondIndex(BaseIndex)`: 增加 `average_duration` 属性。

## 四、附录：指数背景知识

*   **中证指数**: 中证指数有限公司编制了一系列反映中国A股市场的指数，最著名的包括沪深300、中证500等。这些指数是国内市场的重要基准。


这是一个完美的总结性问题，它将我们之前讨论的所有细节提炼成了一个通用的设计哲学。

掌握“何时分开，何时合并”的原则，是构建清晰、可扩展数据模型的关键。下面就是这个原则的完整指南，您可以把它当作您系统的数据模型设计蓝图。

## 5. 模型设计

### 指导原则：如何思考

在做决定时，问自己三个问题：

1.  **核心数据结构是否相同？**
    *   它们是否有几乎一样的核心字段？（例如：股票和ETF都有开高低收、复权价）
    *   这些字段的**单位和业务含义**是否一致？（例如：指数的“点” vs. 股票的“元”）
    *   **规则**：结构相同，可以合并；结构或含义不同，必须分开。

2.  **数据的来源和处理逻辑是否相同？**
    *   它们是否从同一个地方（如交易所行情API）获取？
    *   处理它们（如计算回报率）的逻辑是否一致？
    *   **规则**：来源和逻辑相似，可以合并；来源或逻辑不同，最好分开。

3.  **它描述的是“是什么”（实体）还是“发生了什么”（事件/数据）？**
    *   “是什么” -> 实体模型 (Asset, Fund, Index)
    *   “发生了什么” -> 时间序列数据或行为模型 (AssetMarketDaily, Transaction)
    *   **规则**：不同类型的实体通常需要不同的模型。不同类型的“事件/数据”也需要不同的模型。

### 系统模型设计蓝图

根据以上原则，我们将您理财系统的所有核心数据分为两类：**应该使用同一种模型的（可共享）** 和 **需要单独设计的（应独立）**。

#### A. 应该使用同一种模型的 (Shared Models)

这些模型之所以可以共享，是因为它们代表了同一类事物或行为，具有高度相似的结构和逻辑。

1.  **二级市场日线行情 (`AssetMarketDaily`)**
    *   **使用者**: **股票 (Stock), ETF, LOF, 封闭式基金**
    *   **理由**: 它们都是在交易所进行竞价交易的证券，核心数据结构完全相同（OHLC价格, 成交量, **后复权价**）。数据来源和处理逻辑也完全一致。

2.  **交易流水 (`Transaction`) 和 持仓快照 (`Position`)**
    *   **使用者**: **所有可交易资产 (股票, 基金, 债券等)**
    *   **理由**: “买入/卖出”这个行为，以及“持有”这个状态，对于任何资产来说，其核心记录要素（数量, 价格, 日期, 费用）都是统一的。它们应该关联到通用的 `Asset` 基类。

3.  **关注列表项 (`WatchlistItem`)**
    *   **使用者**: **所有可被关注的实体 (用户 <-> 资产/指数)**
    *   **理由**: “关注”这个关系本身是通用的。它描述的是用户与一个金融实体之间的联系，与该实体具体是什么类型无关。

#### B. 需要单独设计的 (Separate Models)

这些模型之所以必须独立，是因为它们的数据结构、业务含义或来源有着本质的区别。

1.  **指数日线数据 (`IndexDaily`)**
    *   **使用者**: **指数 (Index)**
    *   **理由**: 核心单位是“点位”而非“价格”；没有“复权价”的概念；包含指数整体估值（PE/PB）等特有字段。与任何可交易资产的行情数据都完全不同。

2.  **基金净值数据 (`FundNavDaily`)**
    *   **使用者**: **所有基金 (ETF, LOF, 场外基金)**
    *   **理由**: 这是基金独有的内在价值数据，来源于基金公司每日公告，而非交易所实时行情。其核心字段是 `NAV` 和 `累计NAV`，与市场行情结构完全不同。一个ETF会同时拥有**市场行情数据**和**净值数据**，它们分别存放在 `AssetMarketDaily` 和 `FundNavDaily` 中。

3.  **债券日线数据 (`BondDaily`)**
    *   **使用者**: **债券 (Bond)**
    *   **理由**: 其核心日线数据不是价格，而是**到期收益率 (YTM)、久期、凸性**等衍生指标。这些指标的结构和计算逻辑与股票、基金完全不同。

4.  **各类资产的实体模型 (Entity Models)**
    *   **`Asset` (基类)**: 提供ID、代码、名称等最顶层抽象。
    *   **`Index` (独立)**: 指数不是资产，是基准，拥有发布商、基准日等独特属性。
    *   **`Fund` (子类)**: 作为**所有基金的集合**，拥有基金公司、费率、投资策略等通用基金属性。
    *   **`Stock` (子类)**: 拥有行业分类、股本等股票特有属性。
    *   **`Bond` (子类)**: 拥有票面利率、到期日、信用评级等债券特有属性。

### 总结表格：您的系统模型速查手册

| 数据类型 | 涉及的资产 | 模型方案 | 核心理由 |
| :--- | :--- | :--- | :--- |
| **二级市场行情** | 股票, ETF, LOF | **共享** (`AssetMarketDaily`) | 结构和逻辑完全相同 (OHLC, 复权价) |
| **基金净值** | 所有基金 | **独立** (`FundNavDaily`) | 结构独特 (NAV, 累计NAV)，来源不同 |
| **指数点位** | 指数 | **独立** (`IndexDaily`) | 单位和含义完全不同 (点位 vs. 价格) |
| **债券指标** | 债券 | **独立** (`BondDaily`) | 核心数据为收益率、久期，而非价格 |
| **交易/持仓** | 所有可交易资产 | **共享** (`Transaction`, `Position`) | 行为和状态的记录方式是通用的 |
| **实体定义** | 股票, 基金, 债券 | **独立子类** (`Stock`, `Fund`, `Bond`) | 各自拥有完全不同的核心描述性属性 |

这个蓝图为您提供了一个清晰、健壮且经过行业实践检验的数据库结构。遵循这个设计，您的理财系统将在数据的准确性、查询性能和未来扩展性上拥有坚实的基础。
