# Snowball 轻量版分支改造方案

## 一、背景

当前项目已经具备较完整的后端能力，包括：

- Flask Web API
- SQLAlchemy 多库绑定
- APScheduler 定时任务
- Redis 缓存
- Dramatiq 异步队列
- xalpha / DataBox 数据拉取与缓存
- Flask-Profiler、迁移脚本、测试夹具等配套设施

这套架构对“功能完整性”是有帮助的，但对“单人本地长期使用”来说，复杂度和资源占用都偏高。

结合当前代码结构，项目的复杂度并不只来自 MySQL，本质上还来自以下几个方面：

- 启动链路会初始化多个基础设施
- Redis 不只是缓存，还参与任务与调度链路
- 调度器使用数据库持久化 JobStore
- `xalpha` 当前默认使用 SQL backend
- 测试和配置层对 MySQL 有较强假设

因此，**轻量版的核心目标不是单纯把 MySQL 替换成 SQLite，而是整体收缩系统边界**。

---

## 二、轻量版目标

建议将轻量版定义为“单人、本地、单机、低并发”的使用形态。

### 目标

- 降低常驻内存占用
- 降低运行依赖数量
- 降低启动复杂度
- 保留核心业务能力
- 为后续 SQLite 化创造条件

### 建议保留的能力

- 资产基础管理
- 交易记录管理
- 基础分析能力
- `xalpha` / DataBox 拉数
- 必要的查询类 API

### 建议降级或移除的能力

- Dramatiq 异步队列常驻 worker
- Redis 作为必选依赖
- APScheduler 的数据库持久化能力
- Flask-Profiler
- 面向多环境部署的复杂配置分层
- 非核心的动态调度与恢复能力

---

## 三、总体策略

建议采用 **“先简化架构，再替换数据库”** 的顺序，而不是反过来。

原因如下：

1. 如果先改 SQLite，会立即进入模型兼容、迁移脚本、测试夹具重写等高成本工作。
2. 但即使数据库换掉，只要 Redis / APScheduler / Dramatiq 还保留当前形态，系统复杂度仍然很高。
3. 对轻量版来说，最大的收益来自“减少必选组件”，而不是“先解决所有方言兼容问题”。

因此推荐路线：

**第一阶段：做 Lite 模式，允许不启 Redis / 队列 / 持久化调度器**

**第二阶段：将 `xalpha` 缓存与默认库拆轻**

**第三阶段：评估业务库是否迁移到 SQLite**

---

## 四、推荐改造原则

### 1. 功能开关优先

先为系统增加显式开关，而不是直接删除功能。

建议增加以下配置开关：

- `ENABLE_REDIS`
- `ENABLE_TASK_QUEUE`
- `ENABLE_SCHEDULER`
- `ENABLE_PERSISTENT_JOBSTORE`
- `ENABLE_PROFILER`
- `ENABLE_XALPHA_SQL_CACHE`

这样可以先在一套代码里跑出轻量模式，再决定是否继续清理。

### 2. 单进程优先

轻量版目标应明确为：

- 单机
- 单 Web 进程
- 无常驻后台 worker
- 可接受低并发

### 3. 缓存允许重建

`xalpha` / DataBox 产生的数据，更适合作为“可重建缓存”看待，而不是高价值主数据。

### 4. 先跑通，再收敛

不要一开始追求：

- 与当前版本功能完全等价
- 全量测试无改动通过
- 全模块一起迁 SQLite

轻量版的优先级应该是：

**先可用，再简洁，再兼容。**

## 五、建议分阶段实施

当前状态补充：

- 阶段 1 已完成
- 阶段 2 已完成
- 阶段 3 还没有开始
- 当前仓库还不能认定为“已经完成 SQLite 迁移”

## 阶段 0：建立轻量版分支

建议单开轻量版分支，不在当前主线直接大改。

建议分支命名示例：

- `codex/lite-mode`
- `codex/lightweight-branch`
- `feature/lite-runtime`

### 阶段目标

- 不影响当前主线可用性
- 允许激进收缩依赖
- 方便逐步验证设计是否合适

---

## 阶段 1：先做 LiteConfig 与条件化启动

这是最值得优先做的一步。

### 改造重点

新增一个轻量版配置类，例如：

- `LiteConfig`
- 或 `LocalLiteConfig`

### 重点改造文件

- `web/settings.py`
- `web/__init__.py`
- `web/common/cache/base.py`
- `web/task/__init__.py`
- `web/scheduler/__init__.py`

### 本阶段目标

让应用在以下条件下仍可启动：

- 不连接 Redis
- 不启动 Dramatiq worker
- 不启用持久化 JobStore
- 不启用 profiler

### 预期收益

- 启动链路明显变短
- 常驻依赖减少
- 系统复杂度立刻下降
- 后续 SQLite 化改造范围更清晰

### 推荐设计

#### 1. Redis 改为可选依赖

当前 Redis 既做缓存，也支撑任务与部分调度逻辑。

轻量版可改为：

- 若 `ENABLE_REDIS=False`，则不初始化 Redis 客户端
- 对依赖 Redis 的接口做降级处理
- 将非关键能力改为内存实现或直接禁用

#### 2. Dramatiq 改为可关闭

轻量版建议默认：

- 不启动 worker
- 同步执行轻量任务，或直接移除异步入口

#### 3. APScheduler 改为可关闭或内存模式

轻量版建议：

- 默认关闭动态调度
- 如果保留调度，只使用 `MemoryJobStore`
- 避免使用数据库持久化任务表

---

## 阶段 2：拆轻 `xalpha` / DataBox 缓存策略

这是第二个高收益阶段。

阶段 2 相关文档现在统一归档到 `web/docs/desc/lite_stage2/`。

这里面已经包含：

- 阶段二结果
- blocker 更新
- 验收与收口结论

### 现状

当前 `xalpha` 使用 SQL backend，并把 engine 注入到配置中。

这意味着：

- 默认库承担了较多缓存表
- 数据缓存与业务数据库边界不够清晰

### 轻量版建议

给 `xalpha` 提供两种轻量后端：

#### 方案 A：CSV 缓存

适合极简版：

- 最容易实现
- 可读可删
- 不依赖数据库方言

缺点：

- 查询能力弱
- 文件数量会变多

#### 方案 B：独立 SQLite 缓存库

适合稍强一点的轻量版：

- 将 `xalpha` / DataBox 缓存放入单独 `sqlite` 文件
- 与业务库分离
- 保留一定查询能力

建议文件示例：

- `data/lite_xalpha_cache.db`

### 推荐结论

优先级建议：

1. 先支持 CSV 缓存
2. 再补独立 SQLite 缓存
3. 最后再决定业务主库是否迁 SQLite

---

## 阶段 3：收缩数据库结构

这一阶段才开始认真处理数据库替换。

### 推荐目标

不是一步把所有库合并，而是先明确三类数据：

#### 1. 核心业务数据

例如：

- 资产
- 记录
- 任务
- 分析结果
- 通知日志

#### 2. 可重建缓存数据

例如：

- `xalpha` 拉取数据
- DataBox 中间缓存结果

#### 3. 可放弃或弱化的数据

例如：

- profiler 数据
- 临时调试信息

### 推荐策略

- 核心业务数据：后续再考虑 SQLite
- 缓存数据：允许清空重建
- profiler：轻量版直接移除

---

## 阶段 4：SQLite 化业务库

只有在前面几个阶段跑稳后，才建议进入这一步。

### 需要重点解决的问题

#### 1. `BigInteger` 自增主键

当前大量模型使用 `db.BigInteger`，SQLite 下高风险。

建议方案：

- 封装跨方言主键类型
- MySQL 下编译为 `BigInteger`
- SQLite 下编译为 `Integer`

这样比直接手改所有模型更稳。

#### 2. `CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`

该写法是 MySQL 特有语法。

建议统一改为 ORM 层兼容写法：

- `server_default=func.now()`
- `onupdate=func.now()`

#### 3. SQLite engine 初始化

轻量版若使用 SQLite，需要补齐：

- `PRAGMA foreign_keys=ON`
- `PRAGMA journal_mode=WAL`
- `PRAGMA busy_timeout`

#### 4. MySQL 特有 SQL

需要替换以下行为：

- `SHOW TABLES LIKE ...`
- `show variables like ...`

应改为 SQLAlchemy inspector 或通用实现。

#### 5. 测试夹具

当前测试夹具直接执行：

- `CREATE DATABASE IF NOT EXISTS ...`

SQLite 下需改为：

- 文件数据库
- 或内存数据库
- 或基于 fixture 的临时文件数据库

---

## 六、建议的第一批最小改动

如果目标是快速得到一个“能跑的轻量版”，建议第一批只做以下内容：

### 第一批范围

1. 新增 `LiteConfig`
2. 启动链路条件化
3. Redis 改为可选
4. Dramatiq 改为默认关闭
5. APScheduler 改为关闭或 `MemoryJobStore`
6. `xalpha` 缓存改为 CSV 或独立 SQLite

### 第一批不做

- 不做全量 SQLite 迁移
- 不重写全部迁移历史
- 不追求所有测试立即兼容
- 不保留完整动态任务中心

### 这样做的好处

- 能尽快验证“轻量版是否真的更适合自己”
- 避免直接投入到成本最大的数据库兼容工作
- 降低返工概率

## 八、推荐轻量版形态

建议先实现一个明确的目标版本：

### 轻量版 v1

- 单机运行
- 单进程 Web
- 无 Redis 依赖
- 无 Dramatiq worker
- 无持久化 APScheduler
- `xalpha` 使用 CSV 或独立 SQLite 缓存
- 业务数据暂时仍可先保留 MySQL

这是最务实的第一版。

### 轻量版 v2

- 核心业务库迁 SQLite
- 缓存库独立 SQLite
- 完成主要模型兼容

### 轻量版 v3

- 补测试
- 补导入导出工具
- 补数据库迁移脚本整理

---

## 九、风险与取舍

### 主要风险

- 轻量版与主线能力逐渐分叉
- 若过早迁 SQLite，容易陷入大量兼容修补
- 若保留过多历史能力，轻量版会“名义轻量，实际不轻”

### 建议取舍

轻量版应接受以下现实：

- 部分高级功能不保留
- 某些调度能力改为手动触发
- 某些缓存数据允许丢失
- 与主线并不完全等价

只要核心业务可用，这种取舍就是合理的。

---

## 十、最终建议

建议按以下路线执行：

### 推荐路线

1. 先创建轻量版分支
2. 先做 `LiteConfig` 和启动链路裁剪
3. 先把 Redis / Dramatiq / 持久化调度器变成可选
4. 再拆轻 `xalpha` 缓存
5. 最后再决定业务库是否迁 SQLite

### 不推荐路线

1. 一开始就全量迁 SQLite
2. 一开始就要求轻量版与主线完全等价
3. 一开始就重写所有测试和迁移脚本

## 十二、短期验证（3～5 天）执行计划

这一部分用于指导当前 `codex/lite-spike` 分支的短期验证工作。

目标不是把轻量版一次做完，而是用最小成本回答两个问题：

1. 轻量化是否能显著降低运行依赖和复杂度
2. 这条路线更适合继续留在主仓库推进，还是后续拆成独立项目

### 1. 验证范围

本次短期验证建议只覆盖以下范围：

- Flask Web API 基本启动
- SQLite 下的最小数据库读写
- `xalpha` / DataBox 的最小可用链路
- 1～3 条核心业务接口的冒烟验证

### 2. 本次验证明确不做

为了控制时间和风险，这一轮验证不做以下内容：

- 不追求与主线功能完全等价
- 不迁移全部 MySQL 历史数据
- 不保留 Redis / Dramatiq / 持久化 APScheduler 的完整能力
- 不要求全量测试通过
- 不整理完整 Alembic 迁移历史

### 3. 分支目标

当前短期验证分支建议固定为：

- `codex/lite-spike`

本分支的定位是：

- 可试错
- 可丢弃
- 只为回答架构方向问题服务

不建议将其直接长期演变成“正式轻量版主线”。

### 4. 验收标准

短期验证完成时，至少满足以下 5 个条件：

1. 应用可以在不依赖 MySQL 的前提下启动
2. 轻量模式下不要求 Redis / Dramatiq / 持久化调度器启动成功
3. SQLite 能完成最小核心模型的增删改查
4. `xalpha` / DataBox 至少有一条最小链路可跑通
5. 能明确列出“继续推进的成本”和“阻塞点”

如果以上 5 条达成，本次 spike 就算成功。

### 5. 具体执行顺序

建议按以下顺序推进：

#### 第 1 天：跑通 LiteConfig 与最小启动

重点文件：

- `web/settings.py`
- `web/__init__.py`

目标：

- 新增 `LiteConfig`
- 让应用可以在轻量模式下跳过非核心基础设施初始化
- 明确哪些组件是“可选”，哪些组件仍是“强依赖”

完成标准：

- 应用能启动
- 启动时不会因为 Redis / Dramatiq / profiler / APScheduler JobStore 缺失而直接失败

#### 第 2 天：验证最小数据库能力

重点文件：

- `web/models/__init__.py`
- `web/models/*`
- `web/docs/mysql_to_sqlite_analysis.md`

目标：

- 选最小一组核心模型验证 SQLite 可用性
- 优先处理最容易卡住启动和写入的兼容项

优先修复项：

- `BigInteger` 自增主键
- `CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`
- `TINYINT`
- SQLite engine 初始化参数

完成标准：

- 至少 1～2 个核心模型可以建表、插入、查询

#### 第 3 天：验证 `xalpha` / DataBox 最小链路

重点文件：

- `xalpha/universal.py`
- `xalpha/info.py`
- `web/databox/adapter/data/xa_service.py`
- `web/databox/adapter/data/xa_data_adapter.py`

目标：

- 让 `xalpha` 至少在轻量模式下具备一种稳定后端
- 这轮优先级建议是：
  - 先跑通 SQLite
  - 如果 SQLite 路径成本过高，则退回 CSV 缓存做验证

完成标准：

- 至少一次“写缓存 -> 读缓存”成功
- DataBox 至少一条最小查询链路可用

#### 第 4～5 天：做最小接口冒烟并形成结论

重点文件：

- 核心 router / service
- `web/webtest/conftest.py`
- 新增或调整最小 smoke test

目标：

- 挑 1～3 条最关键的接口做验证
- 输出轻量版是否建议继续推进的结论

建议优先选：

- 资产查询类接口
- 记录查询或写入类接口
- 依赖 DataBox 的一个最小读取接口

完成标准：

- 接口可返回
- 文档里明确记录 blocker、估时和下一步建议

### 6. 验证结束后的判断标准

#### 可以继续在当前仓库推进的信号

- 改动主要集中在配置、启动链路和少量模型兼容
- `xalpha` 兼容只需要小范围补丁
- 大多数核心 service / router 不需要重写
- 主线与轻量版仍能共享大部分代码

#### 应考虑拆成独立项目的信号

- 轻量版需要长期关闭或替换大量基础设施
- 启动入口、配置体系、数据库形态与主线差异持续扩大
- `xalpha` / DataBox / 测试体系都需要独立维护
- 后续发布节奏与主线明显不同

### 7. 本次 spike 的交付物

建议本分支最终至少产出以下内容：

1. 一套可启动的 `LiteConfig`
2. 一份最小 smoke test 或手动验证记录
3. 一份 blocker 清单
4. 一份“是否继续推进”的明确结论

### 8. 推荐结论模板

短期验证结束时，建议用下面格式收尾：

- **结论**：继续 / 暂停 / 拆分项目
- **已验证通过**：列出已跑通能力
- **主要阻塞点**：列出 3～5 个最大问题
- **预计下一阶段工作量**：低 / 中 / 高
- **建议下一步**：继续当前仓库 / 新建独立项目 / 仅保留轻量模式实验

### 9. 本轮 smoke 验证结果（2026-03-19）

本轮按任务 4 的范围补充了最小接口级 smoke test，测试文件为：

- `tests/test_lite_smoke_validation_and_decision.py`

执行命令：

- `python -m pytest -q tests/test_lite_smoke_validation_and_decision.py`

本轮选定的 3 条接口：

- `GET /api/asset/list/`
- `POST /api/record` + `GET /api/record/<id>`
- `GET /token_test/result`

#### 结论

继续。

原因很直接：轻量模式下，应用启动、SQLite 最小读写、核心接口返回、DataBox 最小运行路径都已经有了可重复的自动化验证，当前暴露出来的问题还没有达到“需要拆独立项目”的程度。

#### 已验证通过

- Lite 模式可以在不依赖 Redis、Dramatiq、持久化调度器的前提下启动。
- `GET /api/asset/list/` 可以从 SQLite 返回种子资产数据。
- `POST /api/record` 可以写入交易记录，并自动计算 `transactionsAmount`。
- `GET /api/record/<id>` 可以读回刚写入的交易记录，证明最小读写链路可用。
