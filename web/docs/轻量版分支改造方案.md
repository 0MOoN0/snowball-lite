# Snowball 轻量版分支改造方案

## 当前定位

这份文档现在不再作为“待执行的大规划”，而是作为轻量版改造的总览文档。

重点只保留三类信息：

- 轻量化为什么要做
- 各阶段到底做了什么
- 当前已经走到哪一步、还没走到哪一步

当前项目的轻量化目标，始终不是“只把 MySQL 换成 SQLite”，而是整体收缩系统边界，让项目更适合：

- 单人使用
- 本地运行
- 单机、低并发
- 更少的常驻依赖

## 轻量版目标

轻量版希望保留的核心能力：

- 资产基础管理
- 交易记录管理
- 基础分析能力
- `xalpha` / DataBox 拉数
- 必要的查询类 API

轻量版希望优先降级或移除的部分：

- Redis 作为必选依赖
- Dramatiq 常驻 worker
- APScheduler 的数据库持久化能力
- Flask-Profiler
- 多环境部署下那套较重的运行前提

- lite 已经不只是“最小 happy path”
- 但还不能说“整个仓库的轻量化已经全部完成”
- 也还不能说“全仓库 SQLite 迁移已经完成”

## 阶段 0：建立轻量版分支

这一阶段的核心工作，是把轻量化尝试和原主线隔开，先允许试错。

本阶段完成的事情：

- 明确轻量版改造不直接在原主线硬改
- 先用独立分支和独立仓库基线承接 spike 结果
- 后续把阶段性结果沉淀为 `snowball-lite` 仓库

这一阶段的价值，不在于功能，而在于给后续轻量化改造留出独立空间。

## 阶段 1：LiteConfig 与最小可用链路

这一阶段解决的是“lite 能不能先跑起来”。

本阶段完成的事情：

- 新增 `LiteConfig`
- 把启动链路拆成“数据库是核心依赖，其他基础设施按配置可跳过”
- lite 模式下可以不强依赖 Redis、Dramatiq、持久化 scheduler、profiler
- 建立 SQLite 最小可用链路
- 验证 `xalpha` / DataBox 的最小兼容路径
- 补了最小接口 smoke 验证

阶段一归档文档：

- `web/docs/desc/lite_stage1/01_lite_config_and_bootstrap.md`
- `web/docs/desc/lite_stage1/02_sqlite_minimal_path.md`
- `web/docs/desc/lite_stage1/02_sqlite_blockers.md`
- `web/docs/desc/lite_stage1/03_xalpha_databox_compat.md`
- `web/docs/desc/lite_stage1/04_smoke_validation_and_decision.md`

阶段一结论：

- lite 模式已经可以独立启动
- SQLite 最小链路已经跑通
- 但当时还只是“可行性被证明”，还谈不上稳定扩展

## 阶段 2：真实链路与高风险模型验证

这一阶段解决的是“lite 不能只停留在最小 happy path”。

- 扩大了 SQLite 覆盖面，验证高风险模型而不只最小模型
- 收敛了 lite 测试夹具和运行方式
- 给阶段二做了增量验收和代码审查

阶段二归档文档：

阶段二结论：

- lite 已经明显超过“只会跑最小 happy path”的阶段
- 但仓库层面的完整 SQLite 迁移还没有完成

## 阶段 3：SQLite 兼容基线与受控扩圈

这一阶段解决的是“lite 能不能从局部验证推进到更可信的受控范围”。

本阶段完成的事情：

- 给 `web/webtest` 增加了 stage3 专用 SQLite fixture
- 验证了两条更业务化的 SQLite 集成链路：
  - `AssetService.init_fund_asset_data`
  - `GridTransactionAnalysisService.trade_analysis`
- 收口了阶段三范围内会直接挡路的 MySQL 专有 SQL
- 新增了 `migrations_snowball_lite`
- 引入 `bootstrap_lite_database`，让 lite 不再只依赖 `db.create_all()`
- 对阶段三做了增量验收和代码审查

阶段三结论：

- lite 的 SQLite 可信度已经进一步提升
- 但当前仍然只适合继续做“受控范围内的扩圈”
- 还不适合把结果解读成“全仓库 SQLite 化已经完成”

## 当前结论

到现在为止，轻量化已经完成了很重要的一段，但还没有“全部完成”。

已经完成的部分：

- lite 启动骨架已经稳定
- 核心可选基础设施已经被拆成可关闭
- SQLite 最小可用链路已经稳定
- 真实 DataBox 验证已经补过
- 一批高风险 SQLite 模型已经验证过
- 一批更业务化的 SQLite 集成链路已经验证过
- lite 已经有自己的 migration baseline

还没有完成的部分：

- 整个 `web/webtest` 还没有都脱离 MySQL
- `migrations_snowball_lite` 目前还是受控范围内的 baseline，不是完整 lite schema
- 仓库里剩余 MySQL 专有路径还没有全部系统清理
- 还不能把当前状态认定为“全仓库 SQLite 迁移完成”

## 阶段 4：补齐 lite 主线业务覆盖

这一阶段解决的是“lite 不能只靠局部样例证明自己可用”。

阶段 4 的主目标非常明确：

- 不再只验证最小链路、零散模型和少量样例服务
- 改成按 lite 的核心能力清单补齐 SQLite 下的业务闭环验证
- 开始真正回答“lite 主线能不能脱离 MySQL”

本阶段建议聚焦的能力范围：

- 资产基础管理
- 交易记录管理
- 基础分析能力
- `xalpha` / DataBox 拉数
- 必要的查询类 API

本阶段建议完成的事情：

- 为资产基础管理补齐 SQLite 下的业务闭环验证
- 为交易记录管理补齐 SQLite 下的业务闭环验证
- 为基础分析能力补齐 SQLite 下的业务闭环验证
- 继续补 `xalpha` / DataBox 在 lite 目标范围内的真实或近真实验证
- 为必要查询类 API 补齐 SQLite 集成验证
- 按 lite 目标能力整理一份“已覆盖 / 未覆盖”的能力清单

这一阶段要解决的核心问题是：

- 当前已经验证过的链路，还不足以代表 lite 的完整目标能力
- 如果不先把业务覆盖面补齐，后面即使继续扩 migration，也很难说明 lite 真的能独立使用

阶段 4 的验收标准：

- lite 目标能力都至少有一条明确、可重复的 SQLite 验证路径
- 不再只靠模型级测试或少数样例服务支撑结论
- 能输出一份清楚的 lite 主线能力覆盖清单

阶段 4 的非目标：

- 不要求把整个 `web/webtest` 都迁到 SQLite
- 不要求把所有历史接口都纳入 lite 范围
- 不要求现在就宣称 lite 已完全脱离 MySQL

## 阶段 5：收口 lite 的 SQLite 初始化和运行时

这一阶段解决的是“lite 虽然越来越能跑，但还没有形成正式主线口径”。

阶段 5 的主目标是：

- 把 SQLite 从“已经可跑”推进到“主线正式口径”
- 让 lite 的启动、建库、迁移、运行都不再偷偷依赖 MySQL

本阶段建议完成的事情：

- 扩展 `migrations_snowball_lite`，覆盖 lite 所需完整 schema
- 继续把 lite 主线业务链路纳入统一的 bootstrap / migration 口径
- 清理 lite 主线路径里剩余的 MySQL 专有逻辑
- 继续减少 `db.create_all()` 作为主初始化手段的依赖
- 梳理 lite 启动、初始化、运行时对外部依赖的最后边界

这一阶段要解决的核心问题是：

- `migrations_snowball_lite` 目前还是受控范围内的 baseline，不是完整 lite schema
- 仓库里还有剩余 MySQL 专有逻辑没有按 lite 主线路径彻底收口
- 当前 lite 初始化口径还不够统一，离“正式主线”还差最后一层整理

阶段 5 的验收标准：

- lite 主线路径下，不再需要 MySQL 才能完成初始化和运行
- lite 所需核心 schema 都能通过统一的 migration / bootstrap 路径准备完成
- lite 主线路径里剩余 MySQL 专有逻辑已经基本收口

阶段 5 的非目标：

- 不要求重写 dev/stg/test/prod 的历史迁移
- 不要求现在就把全仓库的数据库方言全部统一
- 不要求现在处理所有非 lite 范围内的旧逻辑

## 阶段 6：零 MySQL 环境收口验收

这一阶段解决的是“前面都做完了以后，怎么证明 lite 主线真的已经脱离 MySQL”。

阶段 6 的主目标是：

- 在没有 MySQL server 的前提下，完成 lite 的启动、初始化、业务验证和最终收口
- 给“lite 主线已经可以纯 SQLite 运行”这个结论补上正式证据

本阶段建议完成的事情：

- 在零 MySQL 环境下启动 lite
- 在零 MySQL 环境下执行 lite 核心回归和业务验收
- 对这一阶段的实现结果做正式代码审查
- 对审查问题做修复、复验和再审查
- 输出最终阶段结论和剩余边界说明

这一阶段要回答的关键问题是：

- lite 是否已经真正不依赖 MySQL
- lite 的核心业务操作是否已经都能在纯 SQLite 环境下完成
- 现在是否可以把 lite 主线认定为“完成脱离 MySQL”

阶段 6 的验收标准：

- 没有 MySQL server 也能完成 lite 启动和初始化
- lite 核心业务操作能在纯 SQLite 环境下完成
- 最终代码审查后没有中等或以上问题残留
- 可以明确输出“lite 主线已脱离 MySQL”或“仍有 blocker”的结论

阶段 6 的非目标：

- 不代替全仓库最终 gate
- 不等于全仓库 SQLite 迁移完成
- 不要求非 lite 配置放弃 MySQL 兼容

## 后续方向

如果后面继续推进，建议按阶段 4、阶段 5、阶段 6 这条线继续往前做。

当前最优先的仍然是：

1. 先做阶段 4，把 lite 目标能力覆盖清楚
2. 再做阶段 5，把 lite 的初始化和运行时正式收口
3. 最后做阶段 6，在零 MySQL 环境下完成集中验收

不建议现在做的事：

- 不建议立刻把全部旧测试统一切到 SQLite
- 不建议现在重写 dev/stg/test/prod 的完整历史迁移
- 不建议把当前结果误读成“轻量化已经全部完成”
- 不建议把“lite 主线脱离 MySQL”和“全仓库 SQLite 化”混成一件事
