# Snowball 系统说明文档

## 概述

Snowball 是一个基于 Flask 的股票数据分析和通知服务系统，采用模块化设计，支持多数据库绑定、定时任务调度和多渠道通知推送。

## 系统架构

Snowball是一个基于Flask框架的现代化Web应用程序，专注于金融数据分析、资产管理和投资策略回测。系统采用微服务化的模块设计理念，各组件之间松耦合，支持多数据库架构，具备高度的可扩展性和可维护性。

### 技术栈

- **Web框架**: Flask 2.2
- **数据库ORM**: SQLAlchemy 1.4 + Flask-SQLAlchemy 3.0.3
- **任务调度**: APScheduler (支持数据库持久化)
- **缓存系统**: Redis
- **异步任务**: Dramatiq消息队列
- **模板引擎**: Jinja2 (用于通知渲染和邮件模板)
- **日志系统**: 定制化日志记录和管理
- **测试框架**: pytest + 自定义测试fixtures
- **数据库迁移**: Flask-Migrate (支持多环境)
- **生产部署**: Gunicorn WSGI服务器 + 自定义日志轮转
- **性能分析**: Flask-Profiler (可选组件)

### 架构特点

- **多数据库架构**: 支持业务数据、分析数据、性能数据的分离存储
- **模块化设计**: 按业务领域划分模块，每个模块包含完整的MVC结构
- **异步处理**: 重计算任务通过消息队列异步执行
- **灵活部署**: 支持开发、测试、生产多环境配置
- **数据驱动**: 基于xalpha量化分析库的金融数据处理能力

## 设计原则

系统设计遵循以下核心原则：

### 1. 一致性优先

- **关键组件强依赖**: 数据库、缓存、调度器等关键组件必须成功初始化，否则应用将终止启动
- **明确的错误处理**: 关键组件失败时直接抛出异常，而非静默降级
- **数据完整性保证**: 优先保证数据的一致性和完整性，而非系统的高可用性
- **事务隔离**: 测试环境采用事务隔离机制，确保测试之间不互相影响

### 2. 优雅降级

- **非关键组件可选**: API文档、性能分析工具等非核心功能失败时不影响应用运行
- **异步任务独立管理**: 消息队列作为可选组件，在Redis连接失败时仍允许应用启动
- **模块化容错**: 各业务模块独立运行，单个模块异常不影响整体系统

### 3. 可扩展性

- **多数据库支持**: 支持业务数据、分析数据、性能数据的分离存储
- **插件化架构**: 通过装饰器和注册机制实现功能的动态扩展
- **异步处理**: 重计算任务通过消息队列异步执行，提升系统响应性能

### 4. 可维护性

- **清晰的分层架构**: Models-Services-Routers三层架构，职责分离
- **统一的异常处理**: 集中的异常处理机制，便于错误追踪和处理
- **完善的日志系统**: 详细的日志记录，支持问题诊断和性能分析
- **自动化测试**: 完整的测试框架，支持单元测试和集成测试

### 5.数据依赖处理

主要是通过工厂方法+适配器+接口（业务逻辑 -> **接口** <- 数据源适配器 -> 外部API）。接口指定能力，适配器处理外部api调用数据，使其返回统一值；通过工厂选择对应的适配器或者替换适配器。案例：

```python
第一步：定义“能力”——创建指标接口 (国标插座)
class MetricFetcherInterface(ABC):
    """
    定义了获取各种指标的“能力”标准。
    任何声称能获取指标的适配器，都必须遵守这个标准。
    """
    @abstractmethod
    def get_dividend_yield(self, symbol: str) -> DividendYieldDTO:
        pass

    @abstractmethod
    def get_pe_ratio(self, symbol: str) -> PeRatioDTO:
        pass
第二步：创建数据源适配器，让它“实现”这个能力 (制造符合标准的变压器并接上插座)
class YahooAdapter(MetricFetcherInterface):
    """
    雅虎财经的适配器。它声称自己有能力获取股息率和市盈率。
    """
    def get_dividend_yield(self, symbol: str) -> DividendYieldDTO:
        print("--- Calling Yahoo API for Dividend Yield ---")
        # ... 调用雅虎API并返回 DividendYieldDTO 的逻辑 ...
        return DividendYieldDTO(symbol=symbol, yield_percentage=1.5)

    def get_pe_ratio(self, symbol: str) -> PeRatioDTO:
        print("--- Calling Yahoo API for P/E Ratio ---")
        # ... 调用雅虎API并返回 PeRatioDTO 的逻辑 ...
        return PeRatioDTO(symbol=symbol, pe_ratio=25.0)
class SomeOtherAdapter(MetricFetcherInterface):
    """
    另一个提供商的适配器。
    假设它只能提供市盈率，不能提供股息率。
    """
    def get_dividend_yield(self, symbol: str) -> DividendYieldDTO:
        # 它没有这个能力，所以应该明确地表示出来
        raise NotImplementedError("SomeOtherProvider 不提供股息率数据")

    def get_pe_ratio(self, symbol: str) -> PeRatioDTO:
        print("--- Calling SomeOtherProvider API for P/E Ratio ---")
        # ... 调用其API并返回 PeRatioDTO 的逻辑 ...
        return PeRatioDTO(symbol=symbol, pe_ratio=28.0)
第三步：业务逻辑只与接口交互
class AssetService:
    def get_asset_pe_ratio(self, asset_id: int, source: DataSource) -> float:
        asset = ... # 获取资产对象

        # 1. 通过工厂获取一个适配器实例。
        #    重要的是，我们知道返回的 provider 对象是一个“国标插座”。
        #    它符合 MetricFetcherInterface 的标准。
        provider: MetricFetcherInterface = get_data_provider_adapter(source)

        # 2. 调用接口定义的方法，完全不用关心背后是哪个适配器在工作。
        #    我们把“热水壶”插进了“插座”。
        pe_ratio_dto = provider.get_pe_ratio(asset.symbol)

        return pe_ratio_dto.pe_ratio
```

#### 接口设计原则

接口设计原则，保持内聚，将单个接口拆分成不同的小接口，由适配器继承。这种情况下会出现新的问题，适配器在实现了很多接口后，可能会变得庞大臃肿。这种情况下有两种处理：

1. 使用垂直排列的方式显示接口，这样看起来更加直观
2. 如果适配器继承接口太多，内部实现过于复杂，可以使用组合的方式，将内部实现拆分成一个个单独的小适配器

案例：

```python
class _YahooPriceWorker(IPriceFetcher):
    """一个只负责价格的内部类"""
    def __init__(self, api_client):
        self._client = api_client

    def get_price(self, symbol: str) -> PriceDTO:
        # ... 专门获取价格的逻辑 ...
        pass

class _YahooDividendWorker(IDividendFetcher):
    """一个只负责股息的内部类"""
    def __init__(self, api_client):
        self._client = api_client

    def get_dividend_yield(self, symbol: str) -> DividendYieldDTO:
        # ... 专门获取股息的逻辑 ...
        pass

# --- 主适配器，现在是一个“门面 (Facade)” ---

class YahooAdapter(IPriceFetcher, IDividendFetcher): # 仍然可以声明它实现了接口
    def __init__(self):
        api_client = self._create_api_client()
        # 它不自己干活，而是持有两个“工人”
        self._price_worker = _YahooPriceWorker(api_client)
        self._dividend_worker = _YahooDividendWorker(api_client)

    # 它将调用委托给内部的工人
    def get_price(self, symbol: str) -> PriceDTO:
        return self._price_worker.get_price(symbol)

    def get_dividend_yield(self, symbol: str) -> DividendYieldDTO:
        return self._dividend_worker.get_dividend_yield(symbol)
```

接口返回值应该由业务决定，并避免返回胖对象。胖对象会导致几个问题：歧义问题，所有值都可能为None，那么接口的存在就没有意义；过多的防御性编程，充斥大量的判断语句；DTO臃肿。

复杂业务功能：可以由多个适配器编排组合，实现更高级的业务功能，尽量保持适配器简单。

## Databox 能力扩展指南

### 能力类型与选择策略

- 指标型能力（数值/轻量DTO）：
  
  - 示例接口：`SecurityIndexValuationInterface`（如股息率）。
  - 选择策略：通过 `AdapterService.get_indicator_adapter(indicator=..., ...)`，使用 `IndicatorEnum` 映射到接口类型。
  - 返回类型：统一为轻量 DTO，不做格式化（遵循“原始数据优先”）。

- 数据集型能力（列表/表格语义）：
  
  - 示例接口：`ConvertibleBondIssuanceInterface`（可转债发行列表）。
  - 选择策略：通过 `AdapterService.get_adapter_by_capability(interface_cls=..., ...)`（能力驱动，不依赖 `IndicatorEnum`）。
  - 返回类型：统一为 DTO 列表，不返回 `DataFrame`。

### 扩展步骤（标准流程）

1. 定义接口（能力）：
   
   - 在 `web/databox/interfaces/` 新增接口，方法返回 DTO 或 DTO 列表；避免胖对象。
   - 示例：`ConvertibleBondIssuanceInterface.list_issuance(start_date, end_date, market) -> List[ConvertibleBondIssuanceData]`。

2. 定义 DTO 模型：
   
   - 在 `web/databox/models/` 新增 `@dataclass`，字段遵循“原始数据优先”，不做格式化。
   - 时间字段使用 `date/datetime`；金额/价格/百分比等按系统规范统一单位（若对外API使用，遵循项目 API 数据单位约定）。

3. 适配器实现：
   
   - 在对应数据源适配器（如 `AkShareAdapter`）实现接口；调用外部 API，将响应映射为 DTO/DTO 列表。
   - 处理异常：网络/字段缺失等统一记录日志并返回空/None；严禁抛未处理异常至业务层。
   - 保持 KISS/YAGNI：只做必要字段映射与最小清洗，避免过度封装。

4. 注册 Provider：
   
   - 在 `web/databox/registry/providers.py` 使用 `register_provider(...)` 注册适配器与元数据：
     - `supported_markets`（如 `{MarketEnum.CN}`）
     - `cost_level`（选择优先级，数值越小越优先）
     - 指标型能力可在 `supported_metrics` 中声明（如 `DIVIDEND_YIELD`）；数据集型能力不必在此列出，走能力驱动。

5. 适配器选择：
   
   - 指标型：`AdapterService.get_indicator_adapter(...)`（严格模式支持 `preferred`）。
   - 数据集型：`AdapterService.get_adapter_by_capability(interface_cls, market=None, preferred=None)`。

6. 服务层封装：
   
   - 在 `web/databox/service/` 新增服务类，统一接入适配器选择并调用接口；严格模式下未匹配能力抛 `WebBaseException`。
   - 仅返回 DTO/DTO 列表，不返回 `DataFrame`。

7. DataBox 门面：
   
   - 在 `web/databox/data_box.py` 新增门面方法：选择适配器 → 委派服务层 → 返回 DTO/DTO 列表。
   - 示例：`get_cb_issuance_list(start_date, end_date, source=None, market=None) -> List[ConvertibleBondIssuanceData]`。

8. 测试与验证：
   
   - 在 `web/webtest/databox/` 添加单测：
     - 使用 `monkeypatch`/mock 外部 API，验证 DTO 映射与能力选择（严格/自动）。
     - 手动触发测试（需外网/安装依赖）通过环境变量控制执行，例如 `RUN_REAL_AKSHARE=1`。

### 命名与错误处理规范

- 命名：接口/DTO/服务/门面按领域直观命名，避免过度抽象。
- 错误处理：统一使用 `weblogger` 记录异常；服务层在严格模式下使用 `WebBaseException` 表达业务错误。
- 响应：保持“原始数据优先”，后端不做格式化；DTO 字段尽量为基础类型（数值/时间/字符串）。

### 何时使用 IndicatorEnum

- 仅当能力属于“指标型”（数值/轻量DTO）且需要通过枚举路由到接口类型时使用。
- 数据集型能力统一采用能力驱动的 `get_adapter_by_capability`，不再加入 `IndicatorEnum`，避免语义混淆。

### 可选增强（按需）

- 缓存：在服务层为稳定数据集增加 Redis 缓存（TTL 适中），减少外部 API 频次。
- 限频：为适配器调用增加最小化限频，保护外部服务。
- 多源编排：通过服务层将多个适配器能力编排为更高级业务功能，保持适配器本身简单。

## 应用启动流程

### 1. 入口点

系统启动有多种方式：

- **直接Python运行**: 通过 `run.py` 脚本 (推荐)
- **Flask命令行方式**: `flask run --no-reload`
- **VSCode调试**: 使用配置好的启动配置
- **生产环境**: 通过 Gunicorn 等 WSGI 服务器

### 2. 初始化过程

应用初始化流程在 `web/__init__.py` 的 `create_app` 函数中定义，分为关键组件和非关键组件两部分：

#### 关键组件（失败时终止启动）

1. **配置加载**: 根据 `SNOW_APP_STATUS` 环境变量加载对应配置
2. **数据库初始化**: 建立多数据库连接，验证连接有效性
3. **模型注册**: 自动发现和注册所有数据模型
4. **缓存系统**: 初始化Redis缓存连接
5. **Flask-RESTX API 初始化**: 创建全局 `Api` 实例和 `Blueprint`
6. **调度器**: 配置和启动APScheduler，支持数据库持久化
7. **路由初始化**: 扫描并注册各个模块的 `Namespace` 到全局 `Api` 实例
8. **API蓝图注册**: 将 Flask-RESTX `Blueprint` 注册到 Flask 应用实例
9. **异常处理器**: 配置全局异常处理机制
10. **业务设置**: 配置xalpha等核心业务组件
11. **会话管理**: 设置数据库会话自动关闭

#### 非关键组件（失败时继续启动）

10. **数据盒子**: 初始化databox数据处理功能
11. **拦截器**: 注册路由拦截器和中间件
12. **跨域设置**: 配置CORS支持
13. **API文档**: 生成Swagger/OpenAPI文档
14. **性能分析**: 初始化性能监控和分析
15. **异步任务队列**: 启动Dramatiq任务处理器
16. **静态资源**: 配置静态文件服务

### 3. 启动检查机制

系统启动时会进行以下检查：

- **数据库连接验证**: 确保所有配置的数据库都可以正常连接
- **Redis连接测试**: 验证缓存系统的可用性
- **调度器状态检查**: 确认APScheduler正常启动
- **重复启动防护**: 防止Flask调试模式导致的重复初始化
- **环境配置验证**: 检查必要的环境变量是否正确设置

### 4. 启动日志

系统启动过程中会记录详细的日志信息：

- 各组件的初始化状态
- 数据库连接信息
- 调度器任务加载情况
- 路由注册统计
- 启动耗时统计

这些日志有助于问题诊断和性能优化。

## Flask-RESTX 路由架构

### API 组织结构

系统使用 Flask-RESTX 构建 RESTful API，采用 Namespace 方式组织路由：

- **analysis**: 数据分析相关API (`/api/analysis`)
- **notice**: 通知服务相关API (`/api/notice`)
- **system**: 系统配置相关API (`/api/system`)

### 核心特性

- **自动参数校验**: 基于模型定义自动校验请求参数
- **统一响应格式**: 所有API返回统一的JSON格式
- **异常处理**: 多层异常处理机制，自动转换为标准错误响应
- **API文档**: 自动生成Swagger UI文档 (`/docs/`)
- **请求/响应序列化**: 自动序列化和反序列化数据

### 架构概述

Snowball 系统基于 Flask-RESTX 构建 RESTful API，提供了完整的 API 文档生成、请求验证、响应序列化等功能。Flask-RESTX 是 Flask-RESTPlus 的社区维护版本，为 Flask 应用提供了强大的 REST API 开发能力。

### 核心组件

#### 1. API 工厂模式

系统采用工厂模式管理 Flask-RESTX 的核心组件：

- **API 工厂**: `web.common.api_factory.py`
  
  - `create_api_blueprint()`: 创建全局 `Api` 实例和 `Blueprint`
  - `get_api()`: 获取全局 `Api` 实例（单例模式）
  - `_setup_error_handlers()`: 配置异常处理器

- **全局 Api 实例**: 确保整个应用中使用同一个 `Api` 实例

#### 5. API数据格式规范

**原始数据优先原则**:

- API接口应返回原始数值数据，不在后端进行格式化处理
- 数据格式化工作由前端负责，确保展示的灵活性和一致性
- 后端专注于数据逻辑处理，前端专注于数据展示

**数据单位标准化**:

- **金额字段**: 以分为单位的数值类型 (如: `maximumOccupancy`, `purchaseAmount`, `presentValue`, `holdingCost`, `dividend`, `profit`)
- **价格字段**: 以毫为单位的数值类型 (如: `unitCost`, `netValue`)
- **百分比字段**: 以万倍为单位的数值类型 (如: `investmentYield`, `turnoverRate`, `irr`, `dividendYield`，15000表示1.5%)
- **份额字段**: 以百倍为单位的数值类型 (如: `attributableShare`，12345表示123.45份)

**Schema设计规范**:

- **API Schema**: 用于API接口，返回原始数值数据，字段名使用驼峰命名
- **VO Schema**: 用于视图展示，进行数据格式化处理，已标记为废弃
- **Domain Schema**: 用于领域模型，字段名使用下划线命名

**API文档要求**:

- 明确标注数据格式为"原始数据"
- 详细说明各字段的数据类型和单位
- 在"关键注意事项"中强调前端格式化责任
- 提供原始数值数据的返回示例

### 常见问题与解决方案

#### 1. Schema 限制

**问题**: Flask-RESTX 的 `attribute` 参数仅支持输出序列化，不支持输入反序列化

**解决方案**: 

- 输入处理：使用 Marshmallow Schema 配合 `data_key` 处理字段名转换
- 输出处理：使用 Flask-RESTX 的 `attribute` 参数进行字段映射

#### 2. 数据类型转换

**问题**: API 字段使用驼峰命名，数据库模型使用下划线命名

**解决方案**:

- 在 Service 层进行字段名转换
- 使用 Marshmallow Schema 进行数据转换
- 定义专门的 DTO (Data Transfer Object) 类

#### 3. 复杂验证

**问题**: Flask-RESTX 内置验证无法满足复杂业务需求

**解决方案**:

- 基础验证：使用 Flask-RESTX 进行格式和类型验证
- 业务验证：在路由处理函数中进行复杂的业务逻辑验证
- 自定义验证器：扩展 Flask-RESTX 的字段类型

5. **内容渲染**: 每个渠道使用对应的渲染策略生成内容
6. **消息发送**: 并行向各渠道发送通知，支持失败重试
7. **状态跟踪**: 记录每个渠道的发送状态和错误信息

- **重试策略**: 失败重试次数和延迟时间

## 最近修复的问题

### 1. 应用重复启动问题

#### 问题描述

发现在某些环境下应用会启动两次，导致以下问题：

- APScheduler被重复初始化
- 定时任务可能被重复执行
- 数据库连接池资源浪费
- 日志中出现重复的启动信息

#### 问题原因

经过分析，主要有两个原因：

1. **Flask调试模式的重载机制**:
   
   - 当`debug=True`时，Flask启动两个进程：监视器进程和工作进程
   - 监视器进程监控文件变化，工作进程运行应用
   - 导致应用实例被创建两次

2. **Windows系统的时区处理问题**:
   
   - Windows对时区的处理与Unix/Linux不同
   - tzlocal库在检查时区时可能报错："Timezone offset does not match system offset"
   - 导致APScheduler在第二次启动时初始化失败

#### 解决方案

1. **防止Flask重复启动**:
   
   - 在`run.py`中设置`debug=False`，避免Flask启动监视器进程
   - 修改VSCode启动配置，移除不必要的参数

2. **避免APScheduler重复初始化**:
   
   - 添加全局变量`_scheduler_initialized`追踪初始化状态
   - 在初始化前检查该变量，避免重复初始化
   - 成功启动调度器后设置该变量为`True`

3. **处理Windows时区问题**:
   
   - 在jobstore初始化时增加对Windows平台的特殊处理
   - 捕获时区不匹配错误，提供明确的日志
   - 必要时降级使用内存存储，保证基本功能可用

### 2. 优化系统一致性

#### 问题描述

系统原先采用了大量try-catch块包裹组件初始化代码，导致即使关键组件初始化失败，应用仍会继续启动。这可能导致运行时出现难以预测的数据不一致问题。

#### 解决方案

1. **区分关键与非关键组件**:
   
   - 明确划分哪些组件对系统一致性至关重要
   - 关键组件包括：数据库、缓存、调度器、路由、异常处理等

2. **移除关键组件的try-catch**:
   
   - 对关键组件初始化不再使用异常捕获
   - 失败时直接抛出异常，终止应用启动

3. **改进初始化流程**:
   
   - 优化初始化顺序，确保依赖关系正确
   - 提供更详细的初始化日志，便于问题诊断

4. **明确的错误提示**:
   
   - 关键组件失败时提供明确的错误信息
   - 通过日志记录详细的初始化过程

## 数据库迁移管理 (Flask-Migrate)

系统使用Flask-Migrate进行数据库版本控制和迁移管理，支持多环境部署和多数据库连接的自动化处理。Flask-Migrate基于Alembic构建，为Flask应用提供了完整的数据库迁移解决方案。

### 迁移系统配置

#### 1. 基础配置

系统在 web/__init__.py 中初始化Flask-Migrate：

- directory参数指定迁移文件存储目录为migrations_snowball_dev

#### 2. 多环境迁移目录

系统采用环境隔离的迁移目录结构：

## 版本与发布管理（Release Please v4）

本项目采用 Release Please v4 进行“语义版本+发布 PR +标签+CHANGELOG”的自动化发布，配合独立的自动合并工作流实现无人值守发布。

- 基线与配置
  
  - 配置文件：`.github/release-please-config.json`（采用 `release-type: simple`，组件名 `snowball`）。
  - 版本清单：`.github/.release-please-manifest.json`（键为路径 `.`，记录当前版本）。
  - 触发分支：`master/main` 上的 `push`。

- 提交规范（Conventional Commits 必备）
  
  - 可触发发布类型：`feat:`（次版本）/`fix:`（修订版本）/`deps:`（依赖升级，修订）。
  - 不触发版本：`docs:`、`chore:`、`style:`、`refactor:`、`test:`、`ci:`、`perf:` 等，默认仅进入 CHANGELOG 分区。
  - 范例：
    - `feat(web): 新增通知内容解析策略`
    - `fix(api): 修复网格监控阈值计算错误`
    - 破坏性变更：`feat(web)!: 调整模板字段（破坏性）`
    - 或在提交正文末尾 footer 写：`BREAKING CHANGE: ...`（必须在 footer 才生效）

- 发布过程
  
  1) `push` 到 `master/main` → 生成/更新发布 PR（标题：`chore(master): release snowball x.y.z`）。
  2) 发布 PR 自动贴 `autorelease: pending` 标签。
  3) 独立工作流 `Auto Merge Release PR` 监听标签/完成事件，满足分支保护后自动合并（`squash`）。
  4) 合并后自动打标签 `vX.Y.Z`、更新 `CHANGELOG.md` 与 GitHub Release。

- 标签与版本
  
  - 默认标签：`vX.Y.Z`；如需组件前缀，可开启 `include-component-in-tag: true` → `snowball-vX.Y.Z`。
  - 若仓库存在“已合并但未打标签”的旧发布 PR，需为对应提交补打标签以解除阻塞（常见日志：`There are untagged, merged release PRs outstanding - aborting`）。

- 强制版本（验证用）
  
  - 单次运行临时指定：在 Action `with:` 中加 `release-as: 1.7.0`，或在提交正文添加 `Release-As: 1.7.0`。

- 常见问题
  
  - 历史非规范提交导致解析报错与耗时增加：按规范提交；必要时在配置里适度设置 `commit-search-depth`、`release-search-depth`。
  - 组件与清单键不一致会回退到 `1.0.0`：使用 `packages: { ".": ... }` 并在 manifest 用键 `"."`。

> 规范要点：始终使用 Conventional Commits；保证触发分支一致；发布 PR 自动合并受分支保护（评审/检查）约束；出现阻塞优先检查标签与历史发布 PR 状态。

### 扩展指引：新增通知业务类型

为方便后续拓展新的通知类型（如“中签公告”“配售缴款提醒”等），建议遵循以下通用流程：

1) 定义业务枚举
   
   - 位置：`web/common/enum/NotificationEnum.py`
   - 在 `NotificationBusinessTypeEnum` 中新增常量（按递增值），例如：`NEW_TYPE = 5`。

2) 模板解析映射
   
   - 位置：`web/services/notice/render/template_resolver.py`
   - 在 `BUSINESS_TYPE_MAPPING` 中添加数字到目录名的映射，例如：`5: 'new_type'`。

3) 分析策略注册（Analysis）
   
   - 位置：`web/services/notice/analyser/notification_analyser.py`
   - 为新业务类型注册至少 BOT、HTML 两种策略（必要时 Markdown），输出字典与模板字段严格对齐（如 `{title, items}`）。
   - 建议在策略层统一日期/数值格式化，模板只负责展示。

4) 渲染策略约定（Render）
   
   - 渲染层已改为“透传分析结果”：`template.render(**parsed_content)`。
   - 新业务不应依赖 `grid_info`；按自身模板契约设计输出结构。

5) 模板新增（Templates）
   
   - 目录结构：`web/template/notifications/<dir>/<render_type>/default.{txt|html}`。
   - `<dir>` 与映射一致（如 `new_type`）；`render_type` 为 `bot` 或 `html`。
   - 如需特定样式，支持通过 `template_key` 指向自定义模板文件。

6) 服务层映射与内容处理（Service）
   
   - 位置：`web/services/notice/notification_service.py`
   - 在 `_convert_business_type_to_string` 中补充新类型字符串（与模板目录同名）。
   - `make_notification`：除 `GRID_TRADE` 外，默认保存原始 `content`，渲染阶段再分析；若新类型确有“入库预分析”需求，再单独开启，避免耦合其他类型。

7) 生成入口（Scheduler/Service）
   
   - 在调度或业务服务中构造内容字典，保持 KISS：只包含模板需要的字段，避免胖对象。
   - 示例：
     - `{title: 'X提醒', items: [{...}, ...]}` 或 `{title: 'X提醒', summary: {...}}`。

8) 渠道配置（Channels）
   
   - 位置：系统设置中的通知配置服务 `notification_config_service`。
   - 为新业务类型设置启用的渠道列表；无启用渠道时将回退到默认渠道。

9) 测试（Tests）
   
   - 单元测试：分析策略输出字段与模板契约；渲染策略透传效果；业务映射正确性。
   - 端到端：构造通知→渲染→发送（可使用 rollback 会话，避免持久化副作用）。
   - 避免依赖环境变量作为跳过条件；外部依赖（如 AkShare）不可用时以跳过处理。
